# Intake Fields — Shared Data Dictionary

> **Purpose.** This file is the single inventory of every piece of information the agent needs from the user to produce a compliant federal unsolicited proposal under FAR Subpart 15.6. It is read by **Mode 2 (Intake Checklist)** as a single structured pass and by **Mode 3 (Full Workflow)** as a per-phase reference. Both modes share this file so the field list can never drift between them.
>
> **Do not edit this file when adapting the kit for a real user.** Read it, collect the answers, then draft into the user's actual proposal. The fields here are universal — every unsolicited proposal needs this information regardless of the agency.

---

## How to use this file

When the agent reads this file, it walks the user through the field groups below in order, asking for each piece of information conversationally — **not as a web form**. The goal is to collect everything the user knows up front so the drafting phase can move quickly without 30 follow-up questions.

### Tag conventions

Every field is tagged with one of three labels:

- **`[REQUIRED]`** — the proposal cannot be drafted without this. If the user doesn't have it, the agent stops and helps them find it (e.g., "Your UEI is on your SAM.gov profile under 'Entity Information'").
- **`[OPTIONAL]`** — useful but not blocking. Skip if the user doesn't have it. Mode 3 may ask twice; Mode 2 accepts a "skip" the first time.
- **`[ATTACH]`** — a document the user should paste into the chat or attach as a file. Examples: pilot data reports, patent filings, key personnel resumes, past performance contract numbers. Always tell the user *why* the document is needed (e.g., "I need the pilot data so the technical approach section can cite real metrics instead of placeholder numbers").

### What the agent does with collected data

After collecting a group, the agent **summarizes back** what the user provided in plain English and asks for confirmation. This prevents transcription errors and gives the user a chance to correct anything before it lands in the draft.

After all groups are collected, the agent **derives the FAR 15.603(c) eligibility verdict** from the Group C answers and announces pass/fail before drafting begins. If the verdict is fail, the agent stops and recommends the correct alternative channel.

## Group A — Company identity *(required for FAR 15.605(a) Part 1 — Basic Information)*

This is the legal identity of the offeror. Every item here lands in the cover page or Section 1 of the proposal. Most users can answer these from memory or from their SAM.gov profile.

| Field | Tag | What it is | Where to find it |
|---|---|---|---|
| Legal company name | `[REQUIRED]` | Exact name on the SAM.gov registration | SAM.gov → Entity Information |
| Mailing address | `[REQUIRED]` | Street, city, state, ZIP | SAM.gov → Core Data |
| Type of organization | `[REQUIRED]` | "For-profit small business," "for-profit large business," "non-profit," etc. | SAM.gov → Business Information |
| State of incorporation | `[REQUIRED]` | The state your LLC or corporation is registered in | Articles of Incorporation |
| Year founded | `[OPTIONAL]` | Year the company was established | Internal records |
| Number of employees | `[OPTIONAL]` | Headcount, full-time equivalent | Internal records |
| CAGE Code | `[REQUIRED]` | Commercial and Government Entity code, 5 alphanumeric characters | SAM.gov → Entity Overview |
| UEI | `[REQUIRED]` | Unique Entity Identifier, 12 alphanumeric characters (replaced DUNS in 2022) | SAM.gov → Entity Information |
| Primary NAICS code | `[REQUIRED]` | The 6-digit NAICS code that best describes your work (e.g., 541512 = Computer Systems Design Services) | SAM.gov → NAICS Codes |
| Size standard for that NAICS | `[REQUIRED]` | Either revenue ($X million) or employee count, depending on NAICS. Determines if you qualify as small | SBA size standards table |
| Small business status | `[REQUIRED]` | "Yes — small under NAICS XXXXXX" or "No — other than small" | Self-certified at SAM.gov |
| 8(a) certification | `[OPTIONAL]` | SBA's 8(a) Business Development Program certification | SBA Certify portal |
| Woman-Owned Small Business (WOSB) | `[OPTIONAL]` | Or Economically Disadvantaged WOSB (EDWOSB) | SBA Certify portal |
| HUBZone certification | `[OPTIONAL]` | Historically Underutilized Business Zone | SBA Certify portal |
| Service-Disabled Veteran-Owned Small Business (SDVOSB) | `[OPTIONAL]` | Department of Veterans Affairs certification | VA Vetbiz portal |
| Other socioeconomic status | `[OPTIONAL]` | Minority-owned, LGBT-owned, etc. — whatever applies | Self-reported |
| Facility security clearance level | `[OPTIONAL]` | Confidential / Secret / Top Secret / TS/SCI — only if relevant to the target agency | DCSA Industrial Security Letter |
| CMMC level | `[OPTIONAL]` | Cybersecurity Maturity Model Certification level (only if pitching DoD) | CMMC accreditation |

**Note on socioeconomic certifications:** these are not just nice-to-haves. Per `guide.md` Section 5.2, several certifications (especially **8(a)** and **WOSB**) give the contracting officer a legal pathway to sole-source a contract below certain dollar thresholds without writing a full Justification & Approval. **Always put these on the cover page** if the user has them.

## Group B — Target agency *(required for FAR 15.604 preliminary contact and FAR 15.605(b) Mission Support Statement)*

The agent uses this group to verify the user picked a real, current target office and isn't accidentally proposing to a program that has already been competed or dismantled.

| Field | Tag | What it is |
|---|---|---|
| Target federal agency | `[REQUIRED]` | The agency name (e.g., "DCSA," "USSOCOM," "GSA Public Buildings Service"). If the user doesn't know, ask them to describe the mission area and the agent will help identify candidates |
| Target mission center / directorate | `[REQUIRED]` | The specific sub-organization within the agency. Example for DCSA: "Counterintelligence and Insider Threat Directorate." This is the home of the actual mission need |
| Why this agency / why this directorate | `[REQUIRED]` | One paragraph explaining why the user's idea fits this specific office's mission. The agent will pressure-test this in Mode 3 |
| Agency strategic priorities cited | `[OPTIONAL]` | Specific quotes or paraphrases from the agency's published strategic plan, budget justification, or public priorities. These go into the Mission Support Statement |
| Prior contacts with the agency | `[REQUIRED]` | List of every government employee the user has talked to about this idea, with name, office, phone, and date. **Required by FAR 15.605(c)(7).** If "none," that's fine — say "none" |
| Preliminary contact done? | `[REQUIRED]` | Has the user emailed the agency's unsolicited proposal POC per FAR 15.604? Yes/No. If No, the agent recommends doing this BEFORE drafting |
| SAM.gov sources-sought check | `[REQUIRED]` `[ATTACH]` | The user must search SAM.gov for any current solicitation, sources-sought, RFI, or draft PWS in the same topic area. If anything comes up, the proposal is dead per FAR 15.603(c)(6). Attach a screenshot or paste the search results |
| Agency-specific submission instructions verified | `[OPTIONAL]` | Has the user checked the agency's website for current unsolicited proposal submission procedures? FAR 15.606 delegates this to each agency |

## Group C — Eligibility answers *(FAR 15.603(c) six-part test — required before any drafting)*

The agent walks the user through these six questions and uses the answers to derive the eligibility verdict. **A "no" on any question kills the proposal.** This group is the eligibility gate for every workflow mode.

| # | Question | Tag |
|---|---|---|
| 1 | Is the idea **innovative and unique** — genuinely new compared to what the agency and other vendors already provide? Be specific about what makes it new (a patent, a proprietary algorithm, a unique dataset, a new technical approach). | `[REQUIRED]` |
| 2 | Was the idea **independently originated and developed** by the user's company, without any government direction, suggestion, or tipoff? | `[REQUIRED]` |
| 3 | Was the proposal **prepared without government supervision, endorsement, direction, or direct involvement**? Specifically: is any current federal employee helping write the document? | `[REQUIRED]` |
| 4 | Does the proposal include **enough technical and cost detail** for an evaluator to actually judge whether the effort is worthwhile? (Mode 3 will push for line-item cost detail; Mode 2 accepts a ROM total at this stage and refines later) | `[REQUIRED]` |
| 5 | Does the proposal **avoid duplicating an existing agency effort**? Has the user researched in-house programs, existing contracts, and grants in the same topic area, and can they explain why their approach is different/better? | `[REQUIRED]` |
| 6 | Is the proposal **NOT an advance proposal for a known requirement**? (Confirmed by the SAM.gov sources-sought check from Group B) | `[REQUIRED]` |

**Verdict logic:** all six must be "yes" for the proposal to qualify under FAR 15.603(c). If any are "no" or "uncertain," the agent stops and either (a) helps the user research the gap, or (b) recommends the alternative channel.

## Group D — Technical substance *(required for FAR 15.605(b) Part 2 — Technical Information)*

This group feeds the heart of the proposal: what the user is actually proposing to do.

| Field | Tag | What it is |
|---|---|---|
| One-sentence description of the product/service | `[REQUIRED]` | "We do X using Y." Plain English |
| What problem does it solve for the agency | `[REQUIRED]` | One paragraph framing the agency's pain point in terms the agency would recognize |
| What is genuinely new about the approach | `[REQUIRED]` | The "innovation moat." Three or fewer specific differentiators. "We're cheaper" is not acceptable — see `guide.md` Section 2 Test 1 |
| Patent or IP status | `[OPTIONAL]` | Issued patent number(s), pending application number(s), trade secret declarations |
| Existing pilot or proof of concept | `[OPTIONAL]` | Yes/No. If yes, when, where, with whom (under NDA if needed), and what the results were |
| Pilot data report | `[OPTIONAL]` `[ATTACH]` | If a pilot exists, attach the report or paste the key metrics. Numbers in the proposal should be traceable to a real source — never invented |
| Technical white papers or publications | `[OPTIONAL]` `[ATTACH]` | Any peer-reviewed papers, conference presentations, or trade publications about the technology. Attach PDFs or links |
| Independent origin story | `[REQUIRED]` | A short narrative explaining when and how the user's company first conceived the idea, what they invested in developing it, and why no government employee directed any of it. This protects against the FAR 15.603(c)(2) "independently originated" test |
| Required government access (data, facilities, personnel) | `[REQUIRED]` | What does the user need from the agency to execute the work? Even "nothing" is a valid answer |
| Required clearances for the work | `[OPTIONAL]` | If the work requires cleared personnel, what clearance level and how many |

## Group E — Key personnel *(required for FAR 15.605(b) Section 6.3.6)*

Every proposal must name the actual humans who will do the work. Anonymous "team" descriptions get rejected. Per `guide.md` Section 11 kill-shot #10, mismatched personnel is a top failure mode.

For **each** of the following roles, collect the same set of fields:

- **Principal Investigator / Chief Scientist** — the technical lead. Required for any R&D or innovation proposal
- **Operational lead / Director of Operations** — the person running day-to-day execution
- **Program Executive / CEO** — the senior accountable officer
- **Program Manager** — the primary interface with the agency contracting officer

**Per-person fields:**

| Field | Tag |
|---|---|
| Full name | `[REQUIRED]` |
| Title | `[REQUIRED]` |
| Highest degree and institution | `[REQUIRED]` |
| Years of relevant experience | `[REQUIRED]` |
| Active security clearance level (if applicable) | `[OPTIONAL]` |
| Percent commitment to this project | `[REQUIRED]` |
| One-paragraph bio focused on relevant experience | `[REQUIRED]` |
| Full resume | `[OPTIONAL]` `[ATTACH]` |

**Skip rule:** if the company is small enough that one person fills multiple roles, that's fine — just say so. If a role doesn't apply to this specific proposal (e.g., no PI on a services proposal), skip it cleanly.

## Group F — Cost model *(required for FAR 15.605(c) Part 3 — Supporting Information)*

A specific number, not a range. With enough breakdown for the contracting officer to see where the money goes. **Cost realism is one of the six FAR 15.606-2 evaluation factors** — under-detailed cost is a top-three rejection reason.

| Field | Tag | What it is |
|---|---|---|
| Total estimated price | `[REQUIRED]` | A specific dollar figure. ROM is fine at this stage |
| Total period of performance | `[REQUIRED]` | In months. Example: "30 months" |
| Direct labor — hours | `[REQUIRED]` | Total labor hours across the period of performance |
| Direct labor — blended rate | `[REQUIRED]` | Or per-role rates if the user prefers |
| Direct labor — total | `[REQUIRED]` | Hours × rate |
| Subcontractor costs | `[OPTIONAL]` | If the user is teaming with academic partners, lab services, etc. |
| Other direct costs (ODCs) | `[OPTIONAL]` | Materials, software licenses, cloud infrastructure, equipment |
| Travel | `[OPTIONAL]` | For agency reviews, site deployments, etc. |
| Indirect cost rate (G&A or overhead) | `[REQUIRED]` | Usually a percentage applied to direct labor |
| Fee | `[OPTIONAL]` | The user's profit margin (typical: 7–10% on FFP) |
| Rate justification | `[OPTIONAL]` `[ATTACH]` | Documentation of how the labor rates were derived (DCAA-approved rates, market comparables, etc.). Required by FAR 15.609(a) for the line-item budget |

**The cost question Mode 3 will push hard on:** "Where did each of these numbers actually come from?" Vague answers get flagged.

## Group G — Past performance *(required for FAR 15.605(c) Section 6.4.5)*

Three to five references. Past performance is one of the six FAR 15.606-2 evaluation factors. Under-investing in this section is a common small-business mistake.

For **each** past performance reference, collect:

| Field | Tag |
|---|---|
| Customer / agency name | `[REQUIRED]` |
| Contract or project number | `[REQUIRED]` |
| Period of performance | `[REQUIRED]` |
| Total dollar value | `[REQUIRED]` |
| One-paragraph description of the work | `[REQUIRED]` |
| Reference contact (name, email, phone) | `[REQUIRED]` |
| Permission to contact this reference | `[REQUIRED]` |
| Federal vs commercial | `[REQUIRED]` |

**No federal past performance?** That's fine for a first-time federal offeror. Provide commercial references with the same level of detail and be explicit: "Umbra Signals has not yet held a prime federal contract. The references below are commercial."

## Group H — Other disclosures *(required for FAR 15.605(c) Sections 6.4.6 and 6.4.7)*

These are the boilerplate disclosures every proposal must include. Most can be answered with a single sentence.

| Field | Tag | What it is |
|---|---|---|
| Organizational conflicts of interest (OCI) | `[REQUIRED]` | Any current or pending engagement that could affect objectivity. "None" is a valid answer — say it explicitly |
| Active security clearances of the company and personnel | `[OPTIONAL]` | What clearances exist today and how new ones will be obtained if needed |
| Environmental impact considerations | `[REQUIRED]` | Almost always "None" for software/services. State it explicitly |
| Other federal agencies receiving this proposal | `[REQUIRED]` | If the user is submitting to multiple agencies in parallel, disclose all of them. Per FAR 15.605(a)(4) — failure to disclose is a fast rejection |
| Related federal funding currently held | `[REQUIRED]` | Any active SBIR, STTR, BAA, OTA, or CSO contract on related work. Disclose contract numbers and dollar values |
| Prior government contacts about this idea | `[REQUIRED]` | Already collected in Group B; the agent re-confirms here for completeness |

## Group I — Timing and contract preferences *(required for FAR 15.605(c) Sections 6.4.2, 6.4.3, 6.4.4)*

| Field | Tag | What it is |
|---|---|---|
| Period of validity | `[REQUIRED]` | How long the offer remains open. **Minimum 6 months per FAR 15.605(c). Recommend 9–12 months** because evaluation often runs longer than 6 months |
| Preferred contract type | `[REQUIRED]` | Firm-Fixed-Price (FFP), Cost-Plus-Fixed-Fee (CPFF), Time & Materials (T&M), or IDIQ. The agent will recommend if the user is unsure |
| Rationale for the contract type | `[OPTIONAL]` | One sentence explaining why this contract type fits the work |
| Earliest desired start date | `[OPTIONAL]` | When the user could realistically begin if a contract is awarded. Helps the agent calibrate the user's expectations against the realistic 9–18 month evaluation timeline |
| Phase breakdown | `[REQUIRED]` | A high-level Phase 1 / Phase 2 / Phase 3 structure with rough month ranges. Example: "Phase 1 (months 1–6): site selection. Phase 2 (months 7–18): pilot deployment. Phase 3 (months 19–24): assessment and transition" |

## When the intake is complete

After all nine groups are collected, the agent does three things in order:

1. **Summarize back** — present a one-page summary of everything collected, organized by group, and ask the user to confirm accuracy
2. **Run the eligibility verdict** — derive pass/fail from Group C answers and announce it. If fail, stop and recommend the alternative channel
3. **Hand off to drafting** — use the collected data to walk through `guide.md` Sections 6, 7, and 8, producing draft sections of the proposal grounded in the user's real information

After drafting is complete, **always invoke `render-output.md`** to ask the user whether they want the final proposal as a Word .docx file or a PDF, and produce the file.
